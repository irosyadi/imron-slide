Stage 5 – Implementation & Configuration Management

Config Control, Baselines, Change Management, Technical Debt, Integration Planning

Capstone Design – Electrical & Computer Engineering

This session zooms in on Stage 5 of the capstone: Implementation. Students are likely already writing code and building hardware; now we teach them to manage that work professionally using configuration management, baselines, change control, and integration planning. Emphasize: lots of student capstones “die” here due to unmanaged changes and chaotic integration.

Learning Objectives – Stage 5

By the end of this session, you will be able to:

  1. Explain what configuration management (CM) means in a capstone project.
  2. Define and use baselines (requirements, design, implementation) in your project.
  3. Apply a simple change control process using change requests and approvals.
  4. Recognize and manage technical debt in both HW and SW.
  5. Plan for integration using a V‑Model mindset.
  6. Relate these concepts to real ECE project examples (embedded systems, RF, power, FPGA, etc.).

These are behavior‑level objectives: students should do CM, not just know definitions. At the end, you can revisit these objectives and ask students to self‑assess where they feel strong/weak.

Where Are We in the V‑Model?

Note

Stage 5 sits at the bottom of the V:

You turn your designs (DCP‑300) into real artifacts (DCP‑400) while maintaining traceability.

Remind them: - Up to now: mostly documents, diagrams, and planning. - Now: code, hardware, prototypes – but you can’t abandon discipline. Stage 5 sets up the success of Stages 6–8 (testing & validation).

Stage 5 in the Course Deliverables

Stage 5 Key Outcomes:

  • Implementation of planned components (CMP-* → IMP-*).
  • All work under configuration management (Git, etc.).
  • Defined implementation baseline at key milestones.
  • Active change control: every major divergence from design/requirements tracked.
  • Updated Requirements Traceability Matrix (RTM).

DCP‑400 – Implementation

Must include:

  • List of implementation artifacts (IMP-* IDs for code, schematics, PCBs, etc.).
  • Build scripts / README for building SW/HW.
  • Change log (approved changes vs original plan).
  • Updated baselines and CM strategy (branches/tags).

Stress: DCP‑400 is more than “we built it.” It documents how it was built, what versions are official, and how those versions relate back to requirements and design.

Configuration Management – What & Why

Configuration Management (CM) is the discipline of:

  • Identifying the configuration items (CIs) that define your system.
  • Controlling changes to those items.
  • Keeping an accurate record of what version is used where.

Typical CIs in ECE Capstones:

  • Source code, HDL (VHDL/Verilog), firmware.
  • Schematics, PCB layouts, BOMs.
  • FPGA bitstreams, compiled binaries.
  • Test scripts, calibration data, FPGA constraints files.
  • Key documents (StRS, SyRS, design, test plans).

Important

Without CM, you don’t know which version was tested or delivered.

Use the classic horror story:

  • Team tests version A.
  • Team demos accidentally with version B that was only half‑tested. They can’t reproduce results because they never tagged the working commit or documented differences.

Concrete CM Example – FPGA‑Based DSP System

ECE Example: An FPGA‑based OFDM transceiver for a custom wireless link.

Configuration Items (CIs):

  • IMP-SW-101: FPGA HDL source (filters, OFDM mod/demod).
  • IMP-SW-102: MATLAB/NumPy models for algorithm verification.
  • IMP-SY-201: RF front‑end schematic and PCB layout.
  • IMP-SY-202: Clock distribution network design and constraints file.
  • IMP-SW-103: RTL testbenches and simulation scripts.

CM Actions:

  • All HDL and models in a Git repository.
  • Major milestones tagged: v0.1-prototype, v1.0-lab-demo.
  • PCB design files archived with version numbers and date.
  • Change requests logged when modifying OFDM parameters (e.g., subcarrier spacing).

Tie this back to traceability: - Each IMP-* artifact links back to CMP-* (e.g., OFDM core component) and REQ-* (e.g., throughput requirement). Emphasize the importance of tagging bitstreams and matching them to the Git revision that produced them.

Baselines – Freezing a Moving Target

A baseline is a formally agreed version of one or more artifacts that serves as a reference for:

  • Further development
  • Testing
  • Change control

Common baselines in your capstone:

  • Requirements Baseline – DCP‑200 approved (no change without CR).
  • Design Baseline – DCP‑300 approved (architecture/components frozen).
  • Implementation Baseline – DCP‑400 milestone (code/hardware version used for tests).

Once a baseline is set:

  • Changes must go through a controlled process.

Note

Baselines reduce chaos: Everyone knows which version is “the official one” at each stage.

Explain that a “baseline” does not mean “can never change.” It means: from now on, changes are intentional, documented, and reviewed, not accidental.

Change Control – Keeping Changes Under Control

Why change control?

  • New insights: requirements were incomplete or wrong.
  • Hardware constraints: chosen component is EOL or behaves differently.
  • Performance issues: design doesn’t meet timing, power, or noise specs.

Minimal Change Control Process for Capstone:

  1. Identify needed change (bug, new requirement, design fix).
  2. Log change request (CR) with:
    • Description
    • Reason / impact
    • Affected IDs (REQ-*, CMP-*, IMP-*, etc.).
  3. Assess impact (performance, schedule, risk).
  4. Decide (approve / reject / defer) as a team (with advisor input).
  5. Implement and update RTM + docs.

Warning

If you change HW/SW or requirements without a CR, your traceability and test results become invalid.

Show a simple CR template in your LMS or repo (spreadsheet, GitHub Issue template). Encourage linking to related Git commits and requirement IDs.

ECE Change Control Example – Embedded Motor Controller

Project: BLDC motor controller for a robotics platform.

  • Requirement:
    • REQ-SY-030: “Controller shall regulate motor speed within ±5% of setpoint under loads from 0–2 Nm.”

Problem discovered in Stage 5:

  • Lab tests show with 2 Nm load, speed error is ±15%.
  • Root cause: chosen MOSFET has higher Rds_on than expected; thermal issues.

Change Control Actions:

  1. CR‑001 logged: propose changing MOSFET to a lower Rds_on device and updating PCB.
  2. Impact analysis:
    • Cost + $10 per controller.
    • PCB re‑spin requires 2 extra weeks.
    • May affect EMI; new tests needed.
  3. Decision: Approve with schedule adjustment; risk accepted.
  4. Update:
    • CMP-SY-050 (power stage) design updated.
    • IMP-SY-105 (PCB files) new version tagged.
    • DCP‑300 and DCP‑400 updated.
    • Tests re‑run; new results linked to updated components.

This illustrates: - Real ECE constraints (thermal, power) causing design change. - How unmanaged changes would break traceability and schedule.

Technical Debt – The “Credit Card” of Engineering

Technical debt = extra work you incur later because you take shortcuts now.

Sources in ECE projects:

  • Quick‑and‑dirty code “just to make it work” with no tests.
  • Poorly labeled wires / undocumented pin mappings.
  • Temporary breadboard hacks instead of final PCB.
  • Incomplete calibration of sensors (“we’ll fix later”).
  • Skipping documentation (“we’ll write it at the end”).

Debt isn’t always bad if:

  • It’s intentional.
  • It’s documented.
  • You have a plan to pay it back.

Note

Technical debt is like a credit card:

Used carefully, it helps.

Ignored, it compounds and crashes your project.

Ask students for examples from prior projects where they “hacked something” and then regretted it weeks later. Link technical debt to integration pain (things break when you combine subsystems).

Managing Technical Debt – ECE Examples

Mixed‑Signal Sensor Front‑End

  • Shortcut:
    • Use a quick, unshielded breadboard layout for analog front‑end.
  • Debt:
    • Unpredictable noise; every test has random artifacts.

Managing:

  • Log debt item: “Replace breadboard with PCB (IMP-SY-201) before final validation.”
  • Reserve time in schedule and integrate into Stage 6 test plan.

Make them see that “we’ll just debug it later” is a decision that should be tracked like any other risk or change. Encourage them to keep a technical debt register alongside their risk register.

Integration Planning – Don’t Leave It to the Last Week

Integration = combining components so they work as a coherent system.

Bad approach: - “We’ll each build our part and plug everything together the week before the demo.”

Better approach:

  • Plan integration in increments:
    • Start with critical paths (e.g., sensor → microcontroller → actuator).
    • Use stubs/mocks for parts not ready yet.
  • Align integration with verification plans:
    • Each integration step tied to specific REQ-* and VER-* tests.
  • Consider interfaces early:
    • Electrical levels, communication protocols, timing, data formats.

Integration steps example:

  1. SENS ↔︎ CTRL (local link).
  2. CTRL ↔︎ UI (basic local UI).
  3. CTRL ↔︎ CLOUD (remote access).
  4. Full system test with all four.

Point out that integration order matters. Encourage teams to identify high‑risk interfaces and integrate those first.

Integration Planning Example – Multi‑Board Power System

Project: Programmable DC power supply with:

  • Control board (MCU + display).
  • Power board (buck/boost converter).
  • Communication interface (USB/CAN).

Integration Plan:

  1. Stage 5 (early):
    • Integrate MCU with display only; verify UI flows.
    • Power board tested separately with dummy loads.
  2. Stage 5 (mid):
    • Integrate controller board and power board:
      • Verify setpoint‑to‑output mapping.
      • Check ADC readings vs true voltage/current.
  3. Stage 5/6 boundary:
    • Add USB/CAN link to PC app.
    • Execute partial system tests (logging, remote setpoint).
  4. Stage 7:
    • Full integration (controller + power + comm + safety interlocks).
    • System verification against REQ-SY-* (accuracy, response time, safety).

Use this to show integration as a planned sequence, not a one‑shot event. Emphasize that each integration step should be accompanied by targeted tests.

Tying Stage 5 Back to Traceability

At Stage 5, your traceability links focus on:

  • From components to implementation artifacts:
    • CMP-* → IMP-*
  • From implementation to upcoming tests:
    • IMP-* → TST-* (planned)

You should know for each IMP-*:

  • Which CMP-* it implements.
  • Which REQ-* ultimately depend on it.

Important

If you can’t map an IMP-* artifact back to a CMP-* and REQ-*, ask: Why are we building this?

Remind them to keep the RTM updated as they implement and refactor. This ensures by Stage 6 they can design tests that cover all implemented requirements.

Minimal CM Setup – What Your Team Should Do This Week

  1. Version Control:

    • Use a shared Git repo (GitHub, GitLab, Bitbucket, self‑hosted).
    • Commit often with meaningful messages referencing IDs (e.g., REQ-SW-012).

  2. Branching & Tagging:

    • main (or master) for stable, tested code.
    • dev and feature branches (e.g., feature-motor-driver).
    • Tag stable milestones (v0.1-prototype, v0.2-integration1).

  3. Change Log & CRs:

    • Track major changes as issues or in a CR spreadsheet.
    • For each: link to requirements, components, commits.

  4. Baselines:

    • Agree as a team what is Baseline 1 for DCP‑400 (e.g., “version used for first integrated test”).

You can show them a sample Git workflow diagram and/or a screenshot of a well‑structured repository. Offer guidelines for commit message format: [CMP-SW-050][REQ-SW-012] Implement basic PID loop

Quick Interactive Exercise

You are working on an IoT‑enabled smart lighting system for a campus building.

In your team / with a neighbor:

  1. List 3 configuration items (CIs) that should be under version control.
  2. Propose one baseline you would like to establish in the next 3 weeks.
  3. Describe one likely technical debt shortcut you might be tempted to take and how you would manage it.

We’ll share a few answers and discuss.

Walk around while they discuss; prompt them with hints: - CI could include firmware, PCB, cloud backend, mobile app, etc. - Baseline: first end‑to‑end demo version, first hardware revision, etc. - Shortcut: skipping proper security for the cloud, using global variables in firmware, etc.

Summary – Stage 5 Takeaways

  1. Configuration Management keeps your system reproducible and understandable.
  2. Baselines are reference points that stabilize requirements, design, and implementation.
  3. Change control prevents hidden, untracked changes from breaking your project.
  4. Technical debt is inevitable; manage it consciously instead of ignoring it.
  5. Integration planning should start early and proceed in planned increments, not last‑minute chaos.
  6. ECE projects (FPGA DSPs, power converters, embedded controllers, IoT systems) need these practices just as much as large industrial systems.

Important

Stage 5 is where your project can either become a controlled engineering effort or descend into a fragile hack. Your CM and integration discipline determines which.

Close by connecting Stage 5 to upcoming Stage 6: Good CM and integration planning make unit testing and debugging easier. Poor CM multiplies effort in later stages. Encourage questions about repositories, tools, and how to structure DCP‑400.

Next Steps for Your Team

Before the next lab:

  1. Set up or review your Git (or other VC) workflow.
  2. Identify and list your configuration items and proposed first baseline.
  3. Create a simple change request template (issue template or spreadsheet).
  4. Note at least two areas where you are carrying technical debt and how you plan to pay it down.

Bring this information to the lab – we’ll review and refine it together.

You can collect their minimal CM plans as a short graded or ungraded exercise. This forces them to concretely apply what they learned and sets better habits for the rest of the project.
