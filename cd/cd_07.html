Verification vs Validation – One More Time

  • Verification – “Did we build it right?”
    • Against requirements and design.
    • Evidence: tests, analyses, inspections, simulations.
  • Validation – “Did we build the right thing?”
    • Against stakeholder needs & intended use.

At Stage 7, we are still doing verification:

  • Verifying the integrated system against system requirements (REQ-*).
  • Step before Stage 8, where actual users and realistic scenarios dominate.

Tip

Think of Stage 7 as:

“Can we prove that the system does everything the requirements document says, under realistic but controlled conditions?”

Stage 8 might involve field trials, user studies, etc. Stage 7 may still happen in a lab but should mimic real workflows and interface behaviors.

29148: System‑Level Verification Process

ISO/IEC/IEEE 29148 defines Verification as confirming that work products meet their specified requirements.

At the system level, that means:

  1. Plan verification of system requirements (REQ-*).
  2. Design system‑level test cases (VER-*) that trace to REQ-*.
  3. Establish the integration test environment (HW, SW, network, tools).
  4. Integrate components into a working system (HW + SW + data).
  5. Execute tests, record results as objective evidence.
  6. Evaluate coverage of requirements and handle non‑conformances.
  7. Update traceability and configuration status.

Differences from Stage 6: - The items under test are no longer isolated modules, but the end‑to‑end system. - Test cases refer primarily to REQ-* IDs, not just CMP-*.

Stress: 29148 doesn’t force particular tools; it requires planned, traceable, repeatable verification.

Component vs System Verification – Quick Contrast

Stage 6 – Unit / Component Verification

  • Scope: Single component in isolation or with simple stubs.
  • Examples:
    • A motor control firmware module.
    • A FIR filter HDL core.
    • A temperature sensor PCB on the bench.
  • Main artifacts:
    • TST-* test cases.
    • DCP‑500A Unit Verification.

Stage 7 – System / Integration Verification

  • Scope: Whole system, all major components integrated.
  • Examples:
    • Motor + controller + UI + comms.
    • Sensor nodes + gateway + backend + dashboard.
  • Main artifacts:
    • VER-* system test cases.
    • DCP‑500B System Verification.

Important

Unit tests ask: “Does this block work?” System tests ask: “Does the whole thing do what it’s supposed to do?”

Use a concrete analogy: - Stage 6: testing a single LEGO piece for correct dimensions. - Stage 7: check the assembled LEGO car drives down the ramp without falling apart.

29148 Activities – Applied to System Verification

System‑level verification activities (adapted from 29148):

  1. Identify verification scope
    • Which system requirements (REQ-*) will be verified in Stage 7.
  2. Define verification objectives
    • For each REQ-*, specify how it will be verified (Test / Analysis / Modeling / Demonstration / Inspection).
  3. Design system test cases (VER-*)
    • Inputs, environment, preconditions, execution steps, expected results.
  4. Prepare integration & test environment
    • Hardware racks, network, lab setups, simulators, test data.
  5. Execute system tests
    • Run tests, collect data, log outcomes.
  1. Evaluate & report
    • Pass/fail per requirement.
    • Coverage statistics (How many REQ-* have at least one VER-*?).
  2. Handle anomalies & regression
    • Record defects, correct at system or component level, and re‑test.

Relate to their projects: - Many teams will run end‑to‑end scenarios: “Sensor reading → Gateway → Backend → Dashboard.” - That is a VER-* test, often verifying multiple requirements at once (latency, accuracy, robustness).

DCP‑500B – System Verification (Overview)

Main Stage 7 deliverable:

  • DCP‑500B – System Verification

Typical contents:

  1. Purpose and Scope – What is being verified at system level.
  2. System Under Test (SUT) – Architecture snapshot, configuration, versions.
  3. Test Environment & Tools – Lab setup, networks, simulators.
  4. System Test Cases (VER-*) – definitions and links to REQ-*.
  5. Execution Results – Summary + key logs and evidence.
  6. Defects / Issues & Resolutions – integration bugs, performance issues.
  7. Traceability & Coverage – REQ-* ↔︎ VER-*.
  8. Readiness for Validation (Stage 8) – Are we ready to show this to real users?

Note

DCP‑500B is the evidence package that your system fulfills its system requirements and is ready for stakeholder evaluation.

You can reuse the structure of test cases from DCP‑500A (TST‑*) but now focus on: - Higher‑level scenarios. - Cross‑component behaviors and metrics like latency, throughput, power, robustness.

Mapping Requirements to System Tests

Interpretation:

  • Each system requirement (REQ-*) must be verified.
  • Some REQ-* can be partially supported by unit tests (TST-*), but the final answer is at system level (VER-*).
  • The RTM should show coverage: every REQ-* has at least one VER-* (unless justified).

Example: - REQ-SY-011: “System shall display current lab temperatures within 10 s of change.” - Unit tests may check sensor sampling and UI refresh separately. - Only a system test can confirm end‑to‑end latency within 10 s.

ECE Example 1 – Smart Lab Sensor Network (System Level)

Project Recap:

  • Wireless sensor nodes in labs.
  • Gateway forwards data to backend.
  • Dashboard displays current values and alerts.

Relevant system requirements:

  • REQ-SY-011: Node measurements visible on dashboard within 30 s.
  • REQ-SY-012: Data loss < 1% over 24 h.
  • REQ-SY-013: System supports at least 20 nodes simultaneously.
  • REQ-SW-020: Alerts raised when temperature > threshold.

System‑level tests (VER-*) might include:

  • VER-SYS-001: End‑to‑end latency from sensor change to dashboard update.
  • VER-SYS-002: 24‑hour soak test for data loss and reliability.
  • VER-SYS-003: Scalability test with 20 simulated or real nodes.
  • VER-SYS-004: Alert behavior when lab overheats (alarm + log).

Highlight: - Unit tests already checked sensor accuracy and firmware packet interval. - System tests now check the whole pipeline under realistic conditions.

Example – VER‑SYS‑001: End‑to‑End Latency

Test Case ID: VER-SYS-001 – End‑to‑End Delay from Sensor to Dashboard

  • Related Requirement(s):
    • REQ-SY-011: “System shall display updated lab temperature on the dashboard within 30 s of a step change.”
  • System Under Test (SUT):
    • N sensor nodes, 1 gateway, backend, dashboard UI.
  • Method: Test (integrated system).

Setup:

  • At least 1 node in demo lab with adjustable local temperature (e.g., using a hot air source).
  • Backend and UI running in standard deployment configuration.
  • PC screen capture or logging of dashboard timestamps.

Procedure:

  1. Wait for system to reach steady state at baseline temperature.
  2. At time t0, apply a clear temperature step (e.g., raise sensor area from 22°C to ~30°C quickly).
  3. Note the time when the node reading changes (based on internal logs or separate instrumentation).
  4. Observe dashboard; record time t_dashboard when the new temperature first appears.
  5. Compute Δt = t_dashboard – t0.
  6. Repeat for 5 trials.

Pass/Fail Criteria:

  • For all trials: Δt ≤ 30 s.

Ask students: - Where can delays creep in? (sampling interval, radio, queueing, backend processing, UI refresh).

This is exactly why system‑level tests are needed; they reveal overall behavior, not just component‑specific correctness.

ECE Example 2 – Motor Drive Platform

Project Recap:

  • Brushless DC motor drive with embedded controller and GUI.

Key system requirements:

  • REQ-SY-030: Motor reaches commanded speed within 1 s, overshoot < 10%.
  • REQ-SY-031: Emergency stop halts motor within 200 ms.
  • REQ-SY-032: Communication fault detection within 500 ms, safe state on error.

System‑level VER-* tests:

  • VER-SYS-030-01: Step speed command from 0 → 2000 RPM, measure speed vs time.
  • VER-SYS-031-01: Trigger E‑stop while spinning at 1500 RPM, measure stop time.
  • VER-SYS-032-01: Simulate CAN link failure and observe safe‑state transition.

Point out: - Stage 6 probably tested the controller algorithm and the power stage separately. - Stage 7 tests the motor, firmware, HW, and interface together, under realistic operating conditions.

Example – VER‑SYS‑031‑01: Emergency Stop

Test Case ID: VER-SYS-031-01 – Emergency Stop Response

  • Related Requirement(s):
    • REQ-SY-031: “Emergency stop shall bring the motor to a halt within 200 ms from activation under any normal operating condition.”
  • SUT:
    • Full motor drive system (motor, power board, controller, E‑stop hardware).
  • Method: Test (physical measurement).

Setup:

  • Motor mounted securely with appropriate load.
  • Current and speed measurement (e.g., encoder + oscilloscope).
  • E‑stop button wired as per design.

Procedure:

  1. Command motor to steady 1500 RPM.
  2. Using a digital IO or scope trigger, activate E‑stop and simultaneously mark t0.
  3. Record motor speed vs time (via encoder or tachometer).
  4. Determine t_stop, when speed reaches ~0 RPM (below a threshold).
  5. Compute Δt = t_stop – t0.
  6. Repeat for speeds 500, 1000, 1500, 2000 RPM (or range defined in DCP‑200).

Pass/Fail Criteria:

  • For all tested speeds, Δt ≤ 200 ms.

Discuss safety: - This test is not only about performance; it has a safety function. - In a real product, such a test might be mandatory for regulatory compliance.

ECE Example 3 – FPGA‑Accelerated Signal Processing Chain

Project Recap:

  • FPGA card performs filtering and decimation; host PC handles control and logging.

System requirements might include:

  • REQ-SY-040: End‑to‑end processing latency ≤ 5 ms.
  • REQ-SY-041: Throughput ≥ 48 k samples/s.
  • REQ-SY-042: Output SNR ≥ 60 dB with specified input.

System‑level VER-* tests:

  • VER-SYS-040-01: Latency test from analog input change to processed data on host.
  • VER-SYS-041-01: Throughput test with continuous data and back‑to‑back frames.
  • VER-SYS-042-01: End‑to‑end SNR test with reference input (e.g., pure tone + noise).

Note: - Stage 6 already verified the FIR core in isolation. - Stage 7: we now test FPGA + ADC + DMA + driver + host software together.

Designing Good System Tests – Key Qualities

Good VER-* system tests are:

  • Requirements‑driven:
    • Explicitly reference REQ-* IDs; not just “general poking around.”
  • Scenario‑based:
    • Reflect realistic use cases or operational scenarios (e.g., 24‑hour operation, network outage, overload).
  • Observable:
    • System outputs and behaviors are measurable and recorded (logs, waveforms, screenshots).
  • Repeatable:
    • Clear setup and procedure to re‑run the same scenario for regression.
  • Cross‑component:
    • Involve multiple subsystems: HW, FW, SW, networking, UI.

Tip

Ask yourself:

“If I change a firmware version, can I re‑run this VER-* test and see if the system still meets the same requirement?”

Encourage teams to prioritize system tests, because they are time‑intensive: - Start with “must‑pass” scenarios that cover critical requirements and risks.

Test Methods (TAMDI) at System Level

As in Stage 6, 29148 supports multiple verification methods:

  • Test – Execute the system and measure behavior (most Stage‑7 work).
  • Analysis – Use models, math, or logs to argue that behavior meets requirements.
  • Modeling/Simulation – Digital twins, network simulators, etc.
  • Demonstration – Qualitative show‑and‑tell (often not sufficient alone).
  • Inspection – Check configuration, documentation, or interfaces manually.

Examples for Stage 7:

  • Use Test to check: latency, throughput, response to faults.
  • Use Analysis to show: worst‑case timing margins from logs.
  • Use Inspection to verify: configuration files match documented settings.

Point out: - Many VER-* will be Test + Analysis: run experiments, then analyze captured data to verify against numbers in REQ-*.

System Test Environment & Configuration

You must define and control the environment in which system tests run.

Example for Smart Lab Sensor Network:

  • Hardware:
    • 5 real sensor nodes + optional simulated nodes.
    • 1 gateway, 1 backend server (lab PC or VM).
    • Router / Wi‑Fi AP if needed.
  • Software:
    • Backend at baseline tag backend-v0.7-stage7.
    • Node firmware node-fw-v1.0-stage7.
    • UI labdash-ui-v1.0-stage7.
  • Network / Infrastructure:
    • Lab VLAN isolated from other networks.
    • Time‑synchronized clocks (NTP) for consistent timestamps.
  • Test Tools:
    • Traffic generator or script to simulate node load.
    • Log collectors, dashboards, metrics exporters.

Note

System verification depends heavily on having a stable, reproducible environment. Document it in DCP‑500B.

Tie to configuration management: - Just as you version your code, you need to version your test environment. - This allows future teams to reproduce your verification results.

DCP‑500B – Example Structure (Short)

  1. Introduction & Scope
    • Which requirements and scenarios are covered by this system verification.
  2. System Under Test (SUT)
    • Architecture overview, versions of HW/SW, configuration.
  3. Test Environment & Tools
    • Lab topology, equipment, simulators, logging.
  4. System Test Cases (VER-*)
    • Detailed definitions: objective, method, setup, procedure, pass/fail criteria.
  5. Test Execution & Results
    • Summary and key evidence.
  6. Defects & Integration Issues
    • Failures, root cause, fixes, retests.
  7. Requirements Coverage & RTM Update
    • How many REQ-* covered by VER-*, any gaps.
  8. Readiness for Validation (Stage 8)
    • Are we ready to put this in front of real stakeholders?

If you already showed DCP‑500A structure, emphasize parallels: - Same pattern, different level: component vs system.

Warning

Failing a system test is common, especially initially. The important thing is: - Capture it, understand it, and use it to improve.

Discuss VER‑SYS‑003 as a mini‑case: - Backend CPU hits 90–100% at 20 nodes. - Might need optimization or a downgraded spec (e.g., 16 nodes for prototype).

Defects and System‑Level Root Cause

When a VER-* test fails, Stage 7 should:

  1. Log the defect with links: VER-*, REQ-*, affected CMP-*/IMP-*.
  2. Analyze the root cause:
    • Integration issue?
    • Single component bug that escaped Stage 6?
    • Misunderstood requirement?
  3. Fix at appropriate level:
    • Could require changes to: firmware, backend, UI, or configuration.
  4. Regression test:
    • Re‑run the failing VER-* and any related tests.

Tip

Use the same issue tracker discipline from Stage 6, but now defects can involve multiple components.

Encourage them to think in terms of “fault tree”: e.g., data loss could be due to: - RF collisions, - gateway buffer overflow, - backend bugs, - database limits.

System‑level tests help locate where to investigate next.

Show them how TST-* (Stage 6) and VER-* (Stage 7) complement each other, not replace each other.

In‑Class Exercise – Draft a System‑Level Test

Activity (5–10 minutes):

  1. Pick one system requirement (REQ-*) from your team’s DCP‑200.
  2. Identify which components participate in fulfilling it.
  3. Draft a system test case (VER-*) that:
    • Has a clear objective.
    • Defines environment / setup.
    • Describes 4–6 procedure steps.
    • States measurable pass/fail criteria.

Examples of good candidates:

  • Latency / response time requirement.
  • Throughput / capacity requirement.
  • Robustness to network loss or power cycling.

Note

We’ll ask a few teams to share their VER-* ideas and map them to REQ-* on the board.

While walking around, check: - They are not just re‑labeling a unit test; it must involve multiple subsystems. - Criteria are numeric or clearly checkable, not just “seems okay”.

Summary – Key Takeaways for Stage 7

  1. Stage 7 – System / Integration Verification checks the integrated system against system requirements (REQ-*), per ISO/IEC/IEEE 29148.
  2. System tests (VER-*) are:
    • Requirements‑driven, scenario‑based, cross‑component, and repeatable.
  3. The main deliverable is DCP‑500B – System Verification, capturing:
    • System under test, environment, VER-* cases, results, defects, and coverage.
  4. Traceability expands: the RTM links REQ-* to VER-* (and also to TST-* where helpful).
  5. A solid Stage 7 greatly reduces risk when you enter Stage 8 – Stakeholder Validation and public demos.

Important

If Stage 6 is about building strong bricks, Stage 7 is about checking that the house built from those bricks is safe, functional, and matches the blueprint.

Close by previewing Stage 8: - After system verification, we’ll bring real stakeholders or surrogate users to validate that the system actually solves their problem.

Invite questions specifically on: - Choosing which requirements to test at system vs unit level. - How to design realistic system test environments in the constraints of the teaching lab. - How deep DCP‑500B needs to be for the course.
